## 1. The Over-fetching Problem

In a REST API, over-fetching occurs when an endpoint returns more data than the client actually needs. A request to `/api/users/123` might return a full user object — name, email, age, address, profile picture, account status, and a dozen other fields — when the client only needed the name and email to render a profile header. The data arrives whether you asked for it or not, because the endpoint's response shape is fixed on the server side.

Under-fetching is the opposite problem. A single endpoint doesn't return enough data to satisfy the client's needs, forcing additional requests. To display a user's profile alongside their recent orders and the products in those orders, a REST client might need to hit `/api/users/123`, then `/api/users/123/orders`, then `/api/products/{id}` for each item — multiple round trips to assemble one view.

GraphQL eliminates both problems at the query level. The client defines exactly which fields it needs in the query itself, and the server returns precisely that shape — nothing more, nothing less. The Hygraph article illustrates this clearly with the burger analogy: REST always gives you the full cheeseburger regardless of what you wanted, while GraphQL lets you specify exactly what goes on it. The Calmops data fetching comparison reinforces this with a side-by-side code example showing four separate REST requests replaced by a single GraphQL query that returns only the requested fields.

## 2. Endpoint and Schema Philosophy

A typical REST API exposes multiple endpoints, each representing a distinct resource — `/users`, `/orders`, `/products`. Each endpoint responds to standard HTTP methods: GET to read, POST to create, PUT or PATCH to update, DELETE to remove. The structure is resource-oriented and predictable, but it also means the server dictates the shape of every response. Clients have no say in what fields come back.

GraphQL takes a fundamentally different approach. The entire API is accessible through a single endpoint — typically `/graphql` — and all requests are sent as POST requests containing a query that describes exactly what data the client wants. As the Medium article puts it, REST is the classic restaurant experience where you get what's on the menu, while GraphQL is the buffet where you take only what you want.

The strongly-typed schema is one of GraphQL's most significant advantages for frontend developers. Every type, field, and relationship in the API is defined upfront in the Schema Definition Language. This means frontend teams know exactly what data is available, what types each field returns, and what's required versus optional — before writing a single line of component code. The Hygraph article notes that tools like GraphQL Code Generator can automatically produce typed query code directly from the schema, catching type mismatches at build time rather than at runtime in production. For a frontend developer, this is the difference between discovering a broken API contract in the console at 2am versus getting a compiler error before the code ever ships.

## 3. Caching and Complexity

REST's caching advantage comes directly from how it uses HTTP. Because each resource lives at a unique URL and is retrieved with a standard GET request, browsers, CDNs, and intermediary proxies can cache those responses automatically using built-in HTTP mechanisms like ETags, Cache-Control headers, and Last-Modified timestamps. The Calmops article demonstrates this with concrete code — a REST endpoint can set `Cache-Control: max-age=3600` and an ETag derived from the resource's last updated timestamp, and the infrastructure handles the rest. No additional configuration required.

GraphQL complicates this significantly. Because all queries are sent as POST requests to a single endpoint — `/graphql` — HTTP's built-in caching infrastructure doesn't apply. Two different queries asking for completely different data look identical at the network level: both are POST requests to the same URL. The cache has no way to differentiate them. As the Medium article explicitly notes, GraphQL has no built-in caching, and teams must implement their own solutions — typically client-side normalized caches like Apollo Client or React Query.

The trade-off becomes worthwhile in applications with complex, interconnected data requirements. A social media dashboard that needs a user's profile, their recent posts, the comments on each post, and the like counts for each comment would require numerous REST requests and significant client-side data assembly. GraphQL can fulfill that entire data requirement in a single query, with the client specifying exactly which fields it needs across all those relationships at once. For data-heavy, relationship-rich applications — the kind where caching complexity is an acceptable cost — GraphQL's flexibility is a meaningful architectural advantage.
